# Investigation Methods - Методы исследования проблем

> Этот файл описывает МЕТОДЫ работы, а не конкретные решения.
> Цель: научить AI-агента правильному подходу к решению проблем.

## 🎯 Главный принцип

**Глубокое исследование за один раз** вместо множества поверхностных попыток.

Лучше потратить больше ресурсов на одну комплексную диагностику, чем делать 5-10 неэффективных итераций.

---

## 📋 Метод 1: Комплексная диагностика внешних API

**Когда применять:** Интеграция с внешним сервисом не работает, ошибка непонятная.

**Алгоритм (выполнить ВСЁ за один раз):**

1. **Проверить секреты:**
   - Lockbox/Secrets Manager - какие ключи там хранятся?
   - Переменные окружения контейнера - что реально передаётся?
   - Совпадают ли ключи в коде и в хранилище?

2. **Проверить формат API:**
   - Прочитать официальную документацию API
   - GET vs POST? Query params vs JSON body?
   - Какие headers требуются?

3. **Проверить конфигурацию сервиса:**
   - Консоль управления сервисом (Yandex Cloud, AWS, etc.)
   - Какие ключи там указаны? Client Key vs Server Key?
   - Какие домены разрешены?

4. **Проверить логи:**
   - Логи контейнера/сервера
   - Что именно отправляется? Что приходит в ответ?

**Пример применения:**
```
❌ Плохо: "Ошибка No secret provided" → попробую другой формат → не работает → ещё попытка...

✅ Хорошо: За один раз проверил:
- Lockbox: ключ есть, начинается с правильного префикса
- Контейнер: переменная передаётся
- Документация: API использует GET с query params, не POST
- Консоль: Client Key и Server Key на месте
→ Нашёл причину: формат запроса неправильный (POST вместо GET)
```

## 📋 Метод 2: Отладка "пустого ответа" gRPC

**Когда применять:** Frontend получает пустой ответ (0 bytes), статус 200.

**Алгоритм:**

1. **Проверить Network tab:**
   - Response size = 0?
   - grpc-status header?
   - grpc-message header?

2. **Проверить backend логи:**
   - Дошёл ли запрос до handler?
   - Какая ошибка в логах?
   - На каком этапе упало?

3. **Проверить цепочку вызовов:**
   - Interceptor → Handler → Service → Storage
   - Где именно ошибка?

4. **Типичные причины:**
   - Метод в publicMethods, но требует auth
   - Ошибка в service, но не залогирована
   - Timeout без обработки

## 📋 Метод 3: Отладка deployment проблем

**Когда применять:** Локально работает, на stage/prod - нет.

**Алгоритм:**

1. **Проверить что задеплоено:**
   - Какая ревизия контейнера активна?
   - Когда был последний деплой?
   - Совпадает ли код?

2. **Проверить конфигурацию:**
   - API Gateway обновлён?
   - Переменные окружения правильные?
   - Секреты доступны?

3. **Проверить логи:**
   - Логи контейнера
   - Логи API Gateway
   - Ошибки при старте?

4. **Типичные причины:**
   - API Gateway не обновлён после деплоя контейнера
   - Секреты не переданы в контейнер
   - Старая ревизия всё ещё активна

## 📋 Метод 4: Исследование перед изменением кода

**Когда применять:** Перед любым изменением кода.

**Алгоритм (см. decision-making.md):**

1. **Прочитать steering файлы:**
   - `decision-making.md` - алгоритм решений
   - `lessons-learned.md` - известные проблемы проекта
   - Релевантные паттерны (backend/frontend)

2. **Понять контекст:**
   - Как это работает сейчас?
   - Какие компоненты затронуты?
   - Есть ли похожие решения в проекте?

3. **Придумать варианты:**
   - Минимум 2-3 варианта решения
   - Оценить каждый по критериям

4. **Самопроверка:**
   - 5 вопросов из decision-making.md
   - Все ответы "да"? → реализовать

## 🧠 Принципы мышления

### 1. Ширина перед глубиной
Сначала собери полную картину, потом углубляйся в детали.

### 2. Один раз, но качественно
Лучше потратить 10 минут на комплексную диагностику, чем 5 раз по 5 минут на поверхностные попытки.

### 3. Документируй метод, не решение
Записывай КАК ты нашёл решение, а не только ЧТО нашёл.

### 4. Проверяй предположения
Не предполагай - проверяй. "Наверное POST" → проверь документацию.

### 5. Логи - твой друг
При любой непонятной ошибке - сначала логи.
