# Decision Making - Алгоритм принятия решений

> Проверенный алгоритм для точных решений с первого раза

## ⚡ АЛГОРИТМ ДЕЙСТВИЙ (читай первым!)

### 📍 Если это issue/ошибка из сканера:

**0. ПОКАЖИ проблемный код** - процитируй строки, которые будешь править

---

### 🔄 Основной алгоритм:

**1. Придумай 2-3 варианта решения** мысленно

**2. Тройная проверка КАЖДОГО варианта:**
- ❓ Это хорошее решение?
- ❓ Оно оптимально и безопасно?
- ❓ Оно элегантно и не избыточно?

**3. Проверь по памяти:**
- `lessons-learned.md` - уроки из разработки
- `security-checklist.md` - принципы безопасности
- `grpc-workflow.md` - шаблоны кода
- `backend-patterns.md` / `frontend-patterns.md` - архитектурные паттерны

**4. Выбери идеальное решение**

**5. ⚠️ ОБЯЗАТЕЛЬНАЯ САМОПРОВЕРКА (5 вопросов) - НЕ ПРОПУСКАЙ:**
- ❓ Действительно ли это решает проблему?
- ❓ Не ухудшил ли я ситуацию?
- ❓ Считал ли я количество операций ДО и ПОСЛЕ?
- ❓ Проверил ли я по документации?
- ❓ Точно ли это лучшее решение?

**6. ⛔ СТОП! Если хотя бы на ОДИН вопрос ответ НЕТ - вернись к шагу 1**

**7. Только после ВСЕХ 5 вопросов реализуй**

---

## 🛑 СТОП-СИГНАЛЫ — Когда ОБЯЗАТЕЛЬНО исследовать ПЕРЕД кодом

### Интеграции с внешними сервисами (YC, AWS, API)
- [ ] Проверил совместимость с нашим стеком (gRPC-Web, Serverless)
- [ ] Прочитал документацию сервиса
- [ ] Нашёл примеры для похожего use case
- [ ] Проверил ограничения (протоколы, форматы данных)

### Новые инструменты/библиотеки
- [ ] Проверил что инструмент поддерживает наш протокол
- [ ] Проверил ограничения и известные проблемы
- [ ] Нашёл примеры использования с gRPC-Web

### Архитектурные решения
- [ ] Проверил совместимость с Serverless (stateless, cold start)
- [ ] Оценил влияние на масштабирование

**⛔ Если хотя бы один чекбокс не отмечен — НЕ ПИШИ КОД, сначала исследуй!**

## 🔄 Перенос контекста между сессиями

**При получении CONTEXT TRANSFER из предыдущей сессии:**

1. **НЕ начинай сразу реализовывать!**
2. Сначала подтверди понимание контекста
3. Спроси пользователя: "Правильно ли я понял, что следующие действия: [описание]?"
4. Дождись подтверждения
5. Только после подтверждения — действуй

## 📊 Примеры применения

### Пример 1: Исправление бага

**Задача:** "Регистрация не работает с именами через дефис"

**Шаг 0: Показать проблемный код**
```typescript
// shared/lib/validation/step1Schema.ts
firstName: yup.string()
  .matches(/^[А-ЯЁа-яё\s]+$/, 'Только кириллица')  // ❌ Проблема здесь
  .required('Обязательное поле')
```

**Шаг 1-2: Варианты решения**
1. Добавить дефис в regex: `/^[А-ЯЁа-яё\s-]+$/`
2. Убрать валидацию совсем
3. Использовать библиотеку для валидации имён

**Тройная проверка варианта 1:**
- ✅ Хорошее? Да, решает проблему
- ✅ Оптимально? Да, простое изменение
- ✅ Элегантно? Да, минимальное изменение

**Шаг 5: Самопроверка**
- ✅ Решает проблему? Да, дефис будет разрешён
- ✅ Не ухудшил? Нет, только расширил допустимые символы
- ✅ Считал операции? Да, regex остаётся O(n)
- ✅ Проверил документацию? Да
- ✅ Лучшее решение? Да, минимальное изменение

**Шаг 7: Реализация**
```typescript
firstName: yup.string()
  .matches(/^[А-ЯЁа-яё\s-]+$/, 'Только кириллица и дефис')
  .required('Обязательное поле')
```

## 💡 Почему это работает

**Проблема:** Делал изменения сразу → исправлял через несколько итераций

**Решение:** Думай ПЕРЕД реализацией → делай правильно с первого раза

**Результат:**
- ✅ Меньше итераций
- ✅ Меньше багов
- ✅ Лучшее качество кода
- ✅ Быстрее разработка
